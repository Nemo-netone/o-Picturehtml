# 文档驱动开发规范 —— 从 ZSSreference (SimulSpeak1) 提炼

> **本文给 AI 阅读。** 内容提炼自参考项目 `ZSSreference`（SimulSpeak1，Go 实时同传系统）的文档体系。
> 目的：让你**理解并复用**这套"文档驱动开发"(Document-Driven Development, DDD) 方法论——
> 不是照抄它的业务，而是学它**用文档组织一个项目**的方式：文档怎么分层、怎么写、怎么和代码/测试/PR 咬合。
>
> 阅读方式：先读 §0 抓精髓，再读 §2 逐个范式（每个都配 ZSSreference 实例），最后用 §4 的清单落地到任何新项目。

---

## 0. 一句话精髓

> **文档是"事实源"(source of truth)，代码是文档的实现。**
> 动手写代码前先读权威文档；文档声明的约束（架构边界、命名、范围、验收）由**测试 / 模板 / 评审**强制落地，而不是靠自觉。

DDD 的反面是"代码先行、文档事后补"——那样的文档会过期、没人信、变成摆设。
ZSSreference 之所以值得学，是因为它的文档**始终是活的、被引用的、被强制的**。

## 1. 文档体系全景（先看清"有哪些零件"）

ZSSreference 的文档不是一堆散落的 md，而是一套有层级、有分工的体系：

| 文件 | 角色 | 为什么好 |
|------|------|----------|
| `README.md`（根） | 项目门面：定位、总览图、结构、依赖声明、快速开始 | 一个文件让任何人 5 分钟看懂"这是什么、怎么跑、原创了什么" |
| `docs/README.md` | 设计文档**索引中枢** + 术语表 | 给文档做导航，定义共享词汇，避免各说各话 |
| `docs/01..05-*.md` | **编号递进**的设计文档 | 需求→架构→前端→后端→接口，依赖顺序即阅读顺序 |
| `CLAUDE.md` | **流程契约**：前置阅读规则 + PR 规范 | 把"该怎么干活"写死，AI/人都照此执行 |
| `docs/frontend/mvp.md` | **执行计划 + 活文档看板** | 范围、里程碑、PR 映射、进度勾选，单一事实源 |
| `docs/frontend/conventions.md` | **约束性规范** | 命名/分层/样式/测试/错误处理的"按什么标准做" |
| `internal/architecture/architecture_test.go` | **架构守护测试** | 把文档声明的架构边界变成会失败的测试 |
| `docs/images/*.dot` + `.svg/.png` | 图表即代码 | 架构图/流程图有可 diff 的源码，不是死图片 |

**关键认知**：文档之间有明确分工，互不重复——
- 「做什么」在 `mvp.md`，「按什么标准做」在 `conventions.md`，「按什么流程提交」在 `CLAUDE.md`。
- 三者互相**引用**而不**复制**（如 conventions 开头写明"PR 流程见 CLAUDE.md，本文不重复"）。

## 2. 七个值得学的范式（逐个 + ZSSreference 实例）

### 2.1 编号递进 + 单一职责的设计文档

`docs/` 用数字前缀把文档排成一条**依赖链**，编号顺序 = 推荐阅读顺序 = 信息依赖顺序：

```
01-requirements  →  02-architecture  →  03-frontend-spec  →  04-backend-spec  →  05-interfaces
   需求是什么         怎么分层设计          前端长什么样          后端怎么实现          接口/协议精确定义
```

每篇只回答一个问题，互不越界。`docs/README.md` 用一段"建议阅读顺序"清单把它们串起来，并对关键篇目（02 架构、05 接口）**加粗**提示重要性。

**学到的**：不要把"需求、架构、接口"塞进一个巨型 README。拆分 + 编号 + 索引，让读者（含 AI）能按依赖顺序逐层进入。

### 2.2 需求文档结构化（FR / NFR / 边界 / MoSCoW / 验收）

`01-requirements.md` 是一份范式级 PRD。它的骨架值得直接复用：

1. **元信息表**（文档版本 / 状态 / 题目 / 交付周期）——文档自带版本与"待评审/已评审"状态。
2. **背景与目标** + **产品定位**（差异化三点）。
3. **核心演示场景（验收主线）**——把"成功长什么样"写成可演示的剧本。
4. **功能需求 FR 表**：`| 编号 | 功能 | 说明 |`，编号 FR1…FR12，全程可被引用。
5. **非功能需求 NFR**：延迟、稳定性、容错降级、可扩展、可靠性。
6. **范围边界**：本期做 / 本期不做（= In / Out）。
7. **MoSCoW 优先级**：Must / Should / Could / Won't——明确取舍。
8. **验收标准**：编号清单，每条可现场核验。

**学到的**：需求要"可编号、可追溯、可验收"。FR 编号会一路贯穿到里程碑、PR、验收，形成追溯链。

### 2.3 架构即测试（最值得学的一点 ★）

文档（`README` / `02-architecture` / `docs/README`）反复声明四层架构的**职责边界**，例如：

> "PBX 不持有同传业务状态；api-server 不直接处理媒体帧。"

光写在文档里，时间一长就会被破坏。ZSSreference 的高明之处：用一个 Go 测试**强制守护**这些边界——`internal/architecture/architecture_test.go` 扫描源码 import，声明"谁不准依赖谁"：

```go
func TestPBXDoesNotDependOnAPIStoreOrLLM(t *testing.T) {
    assertNoImports(t, []string{"internal/pbx", "cmd/pbx-node"}, []string{
        modulePath + "/internal/api-server",  // 媒体面不准依赖业务面
        modulePath + "/internal/store",        // 媒体面不准依赖持久化
        modulePath + "/internal/ai/llm",       // 媒体面不准依赖 LLM
    })
}
```

一旦有人写了越界的 import，`make test` 就会失败。**文档里的架构原则因此变成了"会报错的契约"。**

**学到的**：能被自动验证的规范才不会腐化。架构分层、依赖方向这类"口头君子协定"，尽量落成测试 / lint / CI 检查。

### 2.4 单一事实源 + 交叉引用

每个"数值 / 决策 / 结构"只在**一个**地方定义，其他地方**引用**而非复制：

- 目录结构在 `mvp.md §3` 定义一次，`conventions.md` 写"目录结构以 mvp.md §3 为准"。
- mock 回放序列既是**单元测试的输入**，又是**手动验收的基准**——`conventions.md §5` 明确"同一序列，避免两套口径"。
- 魔法数字收敛到 `config.ts` / `tokens.ts`，文档给出"常量表"（名称 / 含义 / 建议值 / 出处），代码里禁止裸字面量。
- 文档间用**相对链接 + 章节锚**互引，甚至精确到行："对应 mvp 第 145 行风险项"、"见 conventions §5"。

**学到的**：重复 = 不一致的温床。一处定义、多处引用，是文档长期可信的根本。

### 2.5 活文档：进度看板

`mvp.md` 顶部是一个**随时更新的进度看板**，文档本身就是项目状态：

```
**整体进度：2 / 6 里程碑 · 16 / 19 PR**

| 里程碑 | 状态 | PR 进度 |
|--------|------|--------|
| M0 脚手架 | ✅ 已验收 | 2 / 2 |
| M2 字幕面板 | 🟡 进行中 | 3 / 3 |
| M5 边界 + 顶栏 | ⬜ 未开始 | 0 / 2 |
```

子任务用 `- [ ]` / `- [x]` 复选框，状态用统一图例（⬜ 未开始 / 🟡 进行中 / ✅ 已验收）。完成一项就勾一项、更新计数。

**学到的**：让文档承载进度，而不是另开一个工具。复选框 + 计数器让"做到哪了"一目了然，且进 git 历史可追溯。

### 2.6 里程碑 → PR 的可追溯链 + 验收门禁

`mvp.md` 把工作拆成依赖有序的里程碑 M0–M5，**每个里程碑都可独立验证**，且都带一条 `- [ ] **验收**：…` 的硬标准。再往下，§4.1 给出**里程碑 → PR 拆分映射**：

```
**M1 · 协议 + WS 握手（5 / 5）**
- [x] PR-3 `feat(fe): 定义 WS 协议类型与 SubtitleLine`
- [x] PR-4 `feat(fe): WsClient 连接/JSON 编解码/requestId`
...
```

配套的「执行约定」把节奏写死成闭环：

> 开始前切分支、标 `in_progress` → 完成后逐条对照「验收」核验 → 输出状态判断（✅ 通过 / ⚠️ 有遗留 / ❌ 未达标 + 原因）→ 验收通过才以 PR 提交、合并后主分支须可运行、再标 `completed` 进入下一个；未通过则保持 `in_progress` 说明阻塞，**不跨越**。

**学到的**：需求 → 里程碑 → PR → 验收，全链路可追溯，且每步有"通过才放行"的门禁。粒度原则始终是"一个 PR 只做一件事且可运行"。

### 2.7 约束性规范（conventions）

`conventions.md` 回答"按什么标准做"，全部用**表格**承载、可逐条检查：

- **命名表**：组件 PascalCase、Hook `useXxx`、常量 UPPER_SNAKE、WS 事件字符串与协议**逐字一致**；禁拼音、禁无意义缩写、布尔用 `is/has/should`。
- **单向分层**：`config → types → ws/rtc → state → session → components → App`，并写明硬约束（"components 禁止直接 import ws/rtc"）。
- **设计 token**：颜色/动画/阈值集中到 `tokens.ts`，组件内禁写死十六进制和魔法时长。
- **测试要求表**：哪些模块**必须**有自动化测试（纯函数 reducer、类型守卫），哪些手动验收。
- **错误处理对照表**：区分"信令级错误（打断 UI）"与"翻译类静默错误（优雅降级、永不崩主链路）"，每种错误给触发条件 + 是否打断 + 处理文案。
- **日志规范**：统一前缀 `[ws]`/`[rtc]`、分级、**禁止打印 token/凭据/整段音频**。

**学到的**：规范要可执行、可检查、可落到具体文件位置，而不是"请保持代码整洁"这种空话。

## 3. 贯穿全局的元原则（提炼成可迁移的"道"）

1. **单一事实源**：每个事实只定义一次，别处引用。
2. **权威层级 + 冲突上报**：requirements → architecture → spec → mvp(计划) → conventions(标准) → CLAUDE.md(流程)，下层服从上层；需求与权威文档冲突时**先指出冲突并确认，不擅自偏离**。
3. **doc-first（文档先行）**：动手前必读权威文档（CLAUDE.md 把"改前端前必读 mvp.md"设为**强制**）。
4. **表格优先于散文**：结构化信息（FR、决策、风险、命名、错误）一律用表，便于扫读与 diff。
5. **可追溯**：FR → 里程碑 → PR → 验收，编号串联，任何一项都能往上溯源。
6. **强制落地**：能自动验证就别靠自觉——架构用测试守、PR 用模板卡、魔法数字用规范禁、密钥用 .gitignore 拦。
7. **活文档**：进度看板、文档版本/状态字段，让文档随项目一起演进，进 git 历史。
8. **决策留痕**：`mvp.md` 的「关键技术决策」表（决策点 / 选择 / 理由）= 内联的轻量 ADR，解释"为什么这么选"。

## 4. 给 AI 的应用清单（拿到任意项目怎么用）

### 4.1 新项目起步，按此顺序铺文档
1. 写 `docs/01-requirements.md`：元信息表 + FR 表 + NFR + In/Out 边界 + MoSCoW + 验收标准。
2. 写 `docs/02-architecture.md`：分层 + 职责边界 + 数据流；**把"谁不准依赖谁"明确写出来**。
3. 写 `docs/03-frontend-spec.md`：页面目标、布局、组件职责、状态模型、渲染规则、交互流程、体验边界。
4. 写 `docs/04-backend-spec.md`：模块划分、业务编排、核心逻辑、并发/生命周期、错误矩阵、配置、测试策略。
5. 写 `docs/05-interfaces.md`：接口/协议/数据模型/环境变量，作为前后端的对接契约。
6. 写 `docs/README.md`：技术栈表 + 阅读顺序索引 + **术语表**。
7. 写 `CLAUDE.md`：前置阅读规则 + PR 四段式模板 + "主分支恒可运行"。
8. 写执行计划（如 `mvp.md`）：进度看板 + In/Out + 里程碑(带验收) + 里程碑→PR 映射 + 决策表 + 风险表。
9. 写 `conventions.md`：命名 / 单向分层 / token / 具名常量 / 测试要求 / 错误对照 / 日志。
10. 给关键架构边界加一个**守护测试**（import 扫描 / 依赖方向检查）。

### 4.2 接到一个编码任务时
1. **先读**该领域的权威文档（范围 In/Out、目录结构、里程碑顺序）。
2. 若任务含多个功能点，先拆成**多个单功能 PR**的计划。
3. 实现 → 对照「验收」逐条核验 → 输出状态报告（✅/⚠️/❌）。
4. 通过才以 PR 提交（套四段式模板：标题 / 功能描述 / 实现思路 / 测试方式）；更新看板复选框与计数。
5. 需求与文档冲突时，**停下来指出冲突并确认**，不擅自偏离。

### 4.3 写任何文档时的格式约定
- 顶部一句话定位 + （可选）元信息表（版本/状态）。
- 结构化信息用表格；步骤/清单用编号或复选框。
- 跨文档引用用相对链接 + 章节锚（`./conventions.md` `#5`），精确可跳转。
- 一处定义、别处引用；不复制数值与结构。
- 文档末尾交代与其他文档的关系（"本文不重复 X，见 Y"）。

## 6. `ssr` skill 的落地方式（给 AI 执行用）

上面 §1–§5 是“理解版”：让 AI 学会 ZSSreference 好在哪里。真正做项目时，配套的 `Onezzr/ssr` skill 是“执行版”：它把这套方法变成可重复调用的工作流。

| 场景 | `ssr` 应该做什么 | 交付物 |
|------|------------------|--------|
| 新项目起步 | 先建立文档栈，再写代码 | README、`docs/README.md`、`01-requirements`、`02-architecture`、`05-interfaces`、执行计划、conventions、工作规范 |
| 旧项目补规范 | 先做权威地图，找重复、冲突、缺失验收的地方 | 文档事实源表、缺口清单、修订后的单一事实源 |
| 编码任务 | 先读权威文档，确认 In/Out、接口、目录、验收，再实现 | 小粒度改动、验证结果、看板/PR 状态更新 |
| Review / 审计 | 优先查文档与代码是否形成闭环 | README 可读性、FR 追溯、架构边界、接口契约、验收门禁问题 |

`ssr` 的使用原则：

1. **先建权威地图**：需求、架构、接口、计划、规范、流程分别归属哪个文件。
2. **一处定义，多处引用**：同一事实不要复制到多个文档；其他地方用相对链接和章节锚引用。
3. **把约束落成门禁**：凡是写“必须/禁止/不得”，都尽量配测试、lint、CI、PR 模板或验收清单。
4. **只迁移方法，不复制业务**：不要把 SimulSpeak 的同传术语照搬到别的项目；要迁移的是文档结构、追溯链和验证机制。
5. **完成时要能审计**：任何 FR 都能追到里程碑、PR、验收；任何架构边界都能找到守护方式。

如果后续 AI 要直接套用，请优先读取：

- `Onezzr/ssr/SKILL.md`：核心工作流与触发规则。
- `Onezzr/ssr/references/zssreference-patterns.md`：README、需求、架构、接口、执行计划、conventions、工作规范的模板。
- 本文：理解 ZSSreference 为什么这样组织，以及哪些做法最值得迁移。

## 7. 继续优化方向与借鉴来源（这次已补进 `ssr`）

这套 skill 还可以继续优化的关键，不是再写更多抽象原则，而是把“从哪里借鉴、借鉴哪种结构、放到工作流哪个阶段”记清楚。已补进 `ssr` 的重点如下：

| 优化点 | 借鉴来源 | 为什么要特别记住 |
|--------|----------|------------------|
| **README 结构地图** | `ZSSreference/README.md` | 它的目录顺序很适合项目门面：定位、系统总览、数据流、四层架构、项目结构、依赖、快速开始、配置、命令、测试、文档、贡献 |
| **设计文档导航** | `ZSSreference/docs/README.md` | 它不是普通索引，而是文档中枢：技术栈表、架构边界、图表入口、阅读顺序、术语表 |
| **需求规格骨架** | `docs/01-requirements.md` | 元信息、FR/NFR、In/Out、MoSCoW、验收标准，天然形成可追溯需求 |

因此，`ssr` skill 后续使用时应当主动回答三件事：

1. **当前任务处于哪个工作流阶段**：立项、架构、接口、MVP 拆分、规范统一、AI 协作、交付展示，还是 review。
2. **本次借鉴了哪个 ZSSreference 文件**：不要只说“参考 ZSSreference”，要点名具体文件和结构。
3. **最后用什么门禁收尾**：README 可读性、FR 追溯、接口 mock、架构测试、PR 验收、conventions 检查等。

这也是下一步继续强化 `ssr` 的方向：让它不只是“会写文档”，而是能在真实开发节奏里判断“现在该读哪份文档、该产出哪份文档、该用哪条门禁放行”。
