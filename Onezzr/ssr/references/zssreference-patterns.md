特别注意：

- 借鉴 README 时，重点学它的目录顺序和“项目门面”职责，不要只抄快速开始。
- 借鉴 `03-frontend-spec.md` 时，重点学“页面目标 -> 组件职责 -> 状态模型 -> 渲染规则 -> 交互流程”的 UI 规格链路。
- 借鉴 `04-backend-spec.md` 时，重点学“模块划分 -> 业务编排 -> 核心逻辑 -> 生命周期/并发 -> 错误矩阵 -> 测试策略”的服务规格链路。
- 借鉴图表时，重点学“图有源码、源码可 diff、图片可生成”的机制。
- 借鉴执行计划时，重点学“进度看板 + 里程碑验收 + PR 映射”的闭环。
- 借鉴 conventions 时，重点学可检查表格，不写抽象口号。

## README 模板

根 README 要服务“5 分钟理解项目”：

1. 项目名 + 一句话定位。
2. Demo、截图或最小运行入口。
3. 系统总览图或 ASCII 数据流图。
4. 端到端数据流，用编号写清输入如何到输出。
5. 分层架构表：层、组件、职责。
6. 项目结构树，目录注释只写职责。
7. 原创与依赖声明：核心自研能力、第三方库、运行时外部服务。
8. 快速开始：环境要求、安装、启动、访问地址、停止方式。
9. 配置：优先级、关键环境变量、密钥策略。
10. 构建、测试、常用命令。
11. 文档索引：按推荐阅读顺序链接 `docs/01..05` 和补充文档。
12. 贡献规范：单 PR 单功能、主分支可运行、PR 模板位置。

## 架构文档模板

`02-architecture.md` 要讲清“谁负责什么、谁不能碰什么”：

- 项目结构树：目录职责，不写废话。
- 架构图：用户入口、核心服务、外部依赖、存储、异步任务。
- 分层表：层、组件、职责、输入、输出。
- 业务流程：端到端编号步骤。
- 数据流：数据从哪里来，到哪里去，在哪里转换和持久化。
- 生命周期：连接、请求、任务、会话、资源释放。
- 并发与顺序：队列、锁、goroutine/task、超时、取消。
- 容错降级矩阵：故障、影响、降级策略。
- 架构边界：列出禁止依赖关系，准备转成测试。

边界示例：

| 边界 | 规则 | 守护方式 |
|------|------|----------|
| UI 不直接访问网络客户端 | 只能通过 session/store | lint 或 import 扫描 |
| 媒体层不依赖业务层 | 只能上报事件 | import 扫描测试 |
| provider 不持有用户状态 | 只接收 request/context | 单测和接口审查 |

## 前端规格模板

`03-frontend-spec.md` 要把需求和架构翻译成“页面怎么工作”：

- 页面目标：一句话说明这个页面给用户完成什么任务。
- 页面布局：用 ASCII、线框图或截图标注区域，不只写“做一个好看的页面”。
- 组件职责表：组件、职责、输入、输出、状态归属。
- 状态模型：核心 UI state、数据结构、状态流转、谁负责更新。
- 渲染规则：不同消息、数据状态、错误状态如何显示。
- 交互流程：开始、停止、重连、切换、提交、导出等用户动作的顺序。
- 边界体验：空态、加载、失败、权限缺失、长文本、无障碍、响应式。

前端规格不要替代 `05-interfaces.md`，只引用接口契约；它负责说明接口数据进入 UI 后怎么变成状态和渲染。

## 后端规格模板

`04-backend-spec.md` 要把架构边界落到服务实现：

- 模块划分：模块、路径、职责，说明每个模块拥有哪类事实。
- 核心抽象：接口、Provider、Service、Job、Session 等关键类型。
- 策略编排：不同策略怎么分支，默认策略、降级策略是什么。
- 业务接线：一次请求、会话或任务如何经过各模块。
- 核心逻辑：ID 生成、状态维护、队列、幂等、去重、缓存、持久化。
- 并发与生命周期：goroutine/task、锁、channel、取消、超时、资源释放。
- 错误矩阵：故障、影响、降级、是否打断主链路。
- 配置与密钥：环境变量、默认值、敏感性、是否下发前端。
- 测试策略：单测、集成、race/lint/build，以及必须覆盖的降级路径。

后端规格不要替代 `02-architecture.md`；它负责把架构边界细化成可实现、可测试的服务逻辑。

## 接口文档模板

`05-interfaces.md` 是前后端和内部模块的契约：

- 入口地址和鉴权方式。
- 消息/API 总表：方向、type/path、作用。
- 公共字段说明。
- 每个消息/API 的 JSON 示例或 schema。
- 字段表：字段、类型、是否必填、含义。
- 内部接口和数据模型。
- 外部服务 API 映射。
- 环境变量表：变量、默认值、用途、是否敏感。
- 错误约定：哪些错误打断，哪些静默降级。
- 典型流程：用编号或时序图串起来。

## 执行计划模板

执行计划要把“要做什么”变成可推进的活文档：

```markdown
## 0. 进度看板

**整体进度：0 / N 里程碑 · 0 / M PR**

| 里程碑 | 状态 | PR 进度 |
|--------|------|---------|
| M0 脚手架 | ⬜ 未开始 | 0 / 2 |

## 1. MVP 目标（一句话）
## 2. MVP 范围（In / Out）
## 3. 目录结构
## 4. 里程碑（按依赖顺序，每个都可独立验证）
### M0 · ...
- [ ] 子项
- [ ] **验收**：...

## 4.1 里程碑 -> PR 拆分映射
- [ ] PR-1 `type(scope): 单一功能`

## 5. 关键技术决策
| 决策点 | 选择 | 理由 |

## 6. 风险与对策
| 风险 | 影响 | 对策 |

## 7. 验收清单
```

状态规则：

- 只做 In，不顺手做 Out。
- 一个 PR 只做一件事，且合并后主分支可运行。
- 每完成一项立即更新复选框和计数。
- 未通过验收时保持进行中，写明阻塞，不跨越。

## conventions 模板

conventions 回答“按什么标准做”，必须可检查：

| 章节 | 应包含 |
|------|--------|
| 命名 | 文件、类、函数、常量、事件名、布尔命名 |
| 分层 | 目录依赖方向、禁止反向 import、数据单向流 |
| 类型/字段访问 | getter、类型守卫、禁止裸取不稳定字段 |
| 样式/token | 颜色、间距、动画、阈值集中定义 |
| 常量 | 禁止魔法数字，列出名称、含义、出处 |
| 测试 | 哪些模块必须自动化，哪些手动验收 |
| mock | fixture 位置、标准回放序列、复用到测试和手动验收 |
| 环境变量 | `.env.example`、密钥不入库、不打日志 |
| 日志 | 前缀、级别、禁止打印敏感数据 |
| 错误处理 | 错误类型、是否打断 UI、处理文案 |

## 图表即代码规范

ZSSreference 的图表不是一次性截图，而是一套可维护资产：

| 图表来源 | 借鉴点 | 迁移建议 |
|----------|--------|----------|
| `docs/diagrams.md` | 用 Mermaid 保存四层架构、时序图、状态机、流程图、容错图、扩展拓扑 | 文档中直接嵌 Mermaid，便于 GitHub/VS Code 预览 |
| `docs/images/architecture.dot` | 用 DOT 画正式项目架构图，节点是“组件 + 职责”，边是数据/控制流 | 复杂架构图用 DOT，生成 SVG/PNG 放入 README 或 docs |
| `docs/images/business-flow.dot` | 用编号流程讲清用户操作到系统输出的端到端路径 | 交付、答辩、README 展示优先用业务流程图 |

画图规则：

- 每张图只回答一个问题：架构、时序、状态机、流程、容错或扩展，不混在一起。
- 图中的节点要写职责，不只写组件名。
- 边要写数据或控制动作，如 `WebSocket 控制`、`WebRTC 音频上行`、`translation_result`。
- 保留图源码，生成图作为输出物；不要只提交不可 diff 的图片。
- README 放总览图，`docs/diagrams.md` 放完整图集和源码说明。

## 架构守护测试思路

把架构文档中的“禁止依赖”变成自动检查。最通用方式是扫描源码 import：

1. 定义被扫描根目录，如 `internal/pbx`、`cmd/pbx-node`。
2. 定义禁止依赖前缀，如 `internal/api-server`、`internal/store`。
3. 遍历源码文件，发现 forbidden import 就失败。
4. 将该测试纳入常规 test 命令。

迁移到不同语言时使用等价工具：

| 生态 | 可用守护 |
|------|----------|
| Go | `go test` 扫 import |
| TypeScript | ESLint `no-restricted-imports` 或依赖扫描脚本 |
| Java | ArchUnit |
| Python | import-linter |
| 通用 | CI 脚本扫描路径和 import |
