# 项目工作规范

## 前端任务前置规则（强制）

**动手编写或修改任何前端代码之前,必须先完整阅读 `docs/frontend/mvp.md`。**

- 该文档是前端的唯一权威规划:定义了 MVP 范围(In / Out)、目录结构、里程碑 M0–M5 及其验收标准、关键技术决策与风险对策。
- 前端实现必须严格遵循其中的:
  - **In / Out 范围**:只做 In 列表内的事;Out 项(重连/导出/总结/无障碍等)一律后置,不要顺手实现。
  - **目录结构**:文件放置位置与命名以该文档第 3 节为准。
  - **M0–M5 里程碑顺序**:按依赖顺序串行推进,每个里程碑做完先对照「验收」逐条核验,通过才进入下一个(见该文档「执行约定」)。
- 配合阅读 `docs/03-frontend-spec.md`(功能规格)、`docs/05-interfaces.md`(协议接口)。
- 若需求与 `mvp.md` 冲突,先指出冲突并确认,不要擅自偏离。

## PR(Pull Request)提交规范

本项目所有"添加新功能 / 修改功能"的任务,都必须以 PR 形式提交,并遵守以下约定。

### 1. 每个 PR 只做一件事
- 每个 PR 只实现或修改单一功能。
- 鼓励尽可能小、粒度尽可能细的 PR。
- 大功能必须拆分为多个独立 PR 分步提交,不要一个 PR 堆多个改动。
- 动手前,如果发现任务包含多个功能点,先把它拆成多个 PR 的计划,再逐个实现。

### 2. PR 标题与描述必须清晰完整
每个 PR 的描述需包含以下四部分:
1. **标题**:一句话说明本 PR 新增/修改了什么。
2. **功能描述**:说明该功能的作用与使用方式。
3. **实现思路**:简要说明技术选型或核心实现逻辑。
4. **测试方式**:如何验证该功能正常运行。

### 3. 主分支始终可运行
- PR 合并后,主分支代码必须保持可运行状态。
- 评委/任何人在任意时间查看主分支,都应能复现演示效果。
- 因此:不要把半成品、跑不起来的代码合进主分支;每个 PR 自身应是完整、可运行、可演示的最小单元。

### 默认 PR 描述模板(创建 PR 时套用)

```
## 标题
<一句话:本 PR 新增/修改了什么>

## 功能描述
<该功能的作用,以及如何使用>

## 实现思路
<技术选型 / 核心实现逻辑简述>

## 测试方式
<如何验证功能正常运行,例如运行命令、操作步骤、预期结果>
```
